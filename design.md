## Overview

This submission implements a small, self-contained synthetic data generator for GLiNER2. The core goal is to take a short natural language description of a task and emit JSONL-style examples that follow the GLiNER2 training data format, i.e. each line is a dictionary with an `input` field (free-form text) and an `output` field (a structured dictionary). The structured output aggregates four possible task types: NER, classification, relation extraction, and JSON/structured extraction. The key challenges are: inferring which task types are implied by a description, composing multiple types into a single coherent output per example, and enforcing diversity and label balance. All of this must remain simple enough to run on a CPU-only environment while still being easy to extend with richer generators or external models such as a local Ollama-backed LLM.

The implementation is deliberately split into two layers. By default it uses lightweight keyword and pattern heuristics to infer task types and then draws from banks of templates and entities to synthesize examples. On top of that, there is an **optional LLM-backed task inference mode** that calls a local Ollama-served `llama3.2` model to decide which GLiNER2 task types are active and which labels to use. Both paths normalise their decisions into the same `ParsedTask` dataclass, so the rest of the pipeline remains oblivious to how the task inference was performed. In practice this means you can flip between deterministic, rules-only behaviour and a more flexible LLM-powered interpretation without touching any downstream code.

## Task-type inference

Task inference is handled by the private `_infer_tasks` method, which returns a `ParsedTask` dataclass. The dataclass tracks four booleans (`ner`, `classification`, `relation_extraction`, `json_extraction`) and some extra metadata for classification: `classification_task_name` and `classification_labels`. `_infer_tasks` itself is a dispatcher: it either calls a purely rule-based heuristic implementation or an LLM-backed implementation, depending on the `task_inference_mode` provided to the `DataGenerator` constructor.

In **rule-based mode**, the method lowers the task description and runs a sequence of lightweight heuristics:

- For **NER**, it looks for phrases like ‚Äúextract entities‚Äù, ‚Äúextract company names‚Äù, ‚Äúnamed entity‚Äù, or a general pattern such as `extract X names`. These expressions usually appear when the user wants spans like companies, people, or locations.
- For **classification**, it checks for words like ‚Äúclassify‚Äù, ‚Äúclassification‚Äù, ‚Äúsentiment‚Äù, or ‚Äúlabel as‚Äù. Sentiment is special-cased: if the word ‚Äúsentiment‚Äù appears, the task name is set to `"sentiment"` and labels are set to `["positive", "negative", "neutral"]`.
- For **relation extraction**, it checks for ‚Äúrelation‚Äù, ‚Äúrelationship‚Äù, or canonical examples such as ‚Äúworks for‚Äù, ‚Äúlives in‚Äù, or ‚Äúborn in‚Äù. These are typical relation templates in GLiNER-style examples.
- For **JSON/structured extraction**, it looks for ‚Äújson‚Äù, ‚Äústructured‚Äù, ‚Äúkey-value‚Äù, ‚Äúfields:‚Äù, ‚Äúschema‚Äù, or ‚Äúextract as‚Äù, which are all signals that the user cares about structured outputs rather than just spans.

On top of keyword-based detection, the rule-based path attempts to extract classification labels directly from the description. It searches for patterns like ‚Äúinto A, B and C‚Äù or ‚Äúlabels: A, B, C‚Äù and splits the captured text on commas and the word ‚Äúand‚Äù. The logic keeps only short, non-empty labels to avoid spurious matches. If labels are found, they are adopted as the canonical label list for the classification task. If nothing is found, the generator falls back to default labels (`class_a`, `class_b`, `class_c`) so the interface remains usable even for vague descriptions.

In **LLM mode**, `_infer_tasks` makes a single HTTP call to a local Ollama endpoint (configured to use `llama3.2` by default). The prompt explains the four GLiNER2 task types and instructs the model to return a strict JSON object with booleans for each task type plus optional `classification_task_name` and `classification_labels`. The JSON is parsed into a `ParsedTask` instance, and then the same defaulting logic is applied as in the rule-based path (e.g. filling in a generic task name or labels when missing). If anything goes wrong‚Äîno Ollama server running, invalid JSON, or timeouts‚Äîthe LLM call simply returns `None` and `_infer_tasks` falls back automatically to the heuristic implementation. This keeps the LLM integration opt-in and robust.

Finally, after either path has run, the function fills in any missing classification metadata. If a task looks like classification but no task name is provided, it uses `"classification"`. If no labels are provided or extracted, it uses the generic defaults. This design keeps the rest of the code simple: downstream generators never need to worry about `None` values and can always assume a task name and label list exist when classification is active, regardless of whether rule-based or LLM-based inference was used.

## Multi-task composition

The public `generate` method is responsible for composing multi-task outputs. It first infers the tasks from the description and then loops `n` times to generate examples. In each iteration, it accumulates text fragments and output fields:

- If NER is active, it calls `_generate_ner_example`, which returns a `(text, entities)` pair. The text is appended to a list of text segments, and `entities` is attached under `output["entities"]`.
- If classification is active, it calls `_generate_classification_example`, which returns `(text, classifications)` and attaches them under `output["classifications"]`.
- If relation extraction is active, `_generate_relation_example` is used, adding `output["relations"]`.
- If JSON extraction is active, `_generate_json_example` is used, adding `output["json_structures"]`.

At the end of each iteration, the individual text fragments are joined with spaces into the `input` field, while the output dictionary contains one key per active task type. This directly matches the GLiNER2 multi-task JSONL examples documented in the official README: a single `input` with a composite `output` object that may contain `entities`, `classifications`, `relations`, and/or `json_structures`. If, for any reason, no task type is inferred, the generator falls back to a generic classification-only example so the function never returns an empty output.

The multi-task composition is deliberately simple: each sub-generator is independent, and the composition logic only controls how their text fragments are concatenated and how their outputs are merged. This makes the design easy to reason about and test. It also means you can add new task-specific generators (for example, span-level NER with character offsets) without affecting the others.

## Diversity and label balance

The generator enforces diversity and label balance in several ways:

- **Template banks:** Each sub-generator has a bank of templates or value lists that are randomly sampled. For example, `_generate_ner_example` now draws from a larger set of people, companies, and locations and plugs them into several different sentence templates, producing many more surface forms while still guaranteeing valid entities.
- **Random seeds:** The `DataGenerator` accepts an optional `seed`. Internally it uses a dedicated `random.Random` instance, so users can reproduce exact sequences of examples when debugging or comparing different model runs.
- **Rotating labels for classification:** The `_generate_classification_example` method enforces approximate class balance by cycling through labels with the pattern `labels[idx % len(labels)]`. If there are three labels and 30 examples, each label will be used roughly 10 times. This directly addresses the requirement that no label ‚Äúdominates‚Äù the set of examples.
- **Combinatorial sentence construction for classification:** For sentiment-specific tasks, the generator now composes sentences from multiple banks (subjects such as ‚ÄúThe mobile app‚Äù or ‚ÄúCustomer support‚Äù, plus label-specific outcomes such as ‚Äúworked flawlessly and exceeded my expectations‚Äù or ‚Äúkept crashing in the middle of important work‚Äù). This yields a large combinatorial space of unique sentences that are still semantically aligned with the target label. For generic classification tasks, a similar approach combines neutral ‚Äúcontext‚Äù phrases with different domains (support tickets, bug reports, product descriptions) to avoid repetitive boilerplate.

In the notebook‚Äôs fine-tuning section, where we generate training and evaluation JSONL files for a sentiment task, there is an extra layer of diversity control: we explicitly enforce **structural uniqueness** across both splits. A helper function repeatedly samples from `DataGenerator.generate` and hashes each full example (including both `input` and `output`). Examples are accepted only if their hash has not been seen before. Training examples are generated first, and evaluation examples are generated second while reusing the same ‚Äúseen‚Äù set, so an identical example can never appear in both. This directly satisfies the requirement that no example is duplicated within or across the train and eval sets, while still keeping the generation logic simple and composable.

## Architecture decisions and trade-offs

The main architectural choice was to keep the generator as a pure Python module with no third‚Äëparty runtime dependencies beyond the standard library. This keeps the `generate.py` module light-weight, fast to import, and friendly to environments without GPUs or internet access. The only optional integration is the HTTP call to a locally running Ollama server when LLM-based inference is enabled. If no server is available, the system gracefully falls back to the built-in heuristics. All GLiNER2-specific dependencies (the actual model and training API) are contained in the notebook, so users who only need to generate data do not pay the cost of installing or loading a heavy model.

Another design choice was to favour clarity over extreme configurability. The heuristics in the rule-based `_infer_tasks_rules` path are intentionally simple and explicit rather than learned. The label extraction patterns and keyword lists can be read and edited in a few minutes, which is useful for a research engineer who may want to adapt them to a specific domain (for example, biomedical or legal text). The trade-off is that coverage for very unusual phrasings may be limited in pure rule-based mode. Here, the LLM-backed `_infer_tasks_via_llm` path provides a useful escape hatch: users can let `llama3.2` interpret free-form task descriptions more flexibly while still benefiting from the same `ParsedTask` normalisation and downstream generators.

For multi-task composition, we chose to concatenate independent text fragments rather than trying to weave all required annotations into a single tightly coupled narrative. This is much simpler to implement and reason about but can produce slightly contrived sentences where, for example, the sentiment classification text and the NER text are about different topics. From a training perspective this is still acceptable, because GLiNER2 learns to map each `input` to a structured `output` and is not constrained to a human-like narrative structure, but it is a limitation to be aware of. If desired, a future version could also use the local LLM to jointly generate multi-task narratives while still emitting the same structured `output` schema.

## Evaluation and metrics

The notebook evaluates both the base and fine-tuned GLiNER2 models on a held-out synthetic set using two complementary kinds of metrics.

**Classification accuracy** is the fraction of examples where the predicted label exactly matches the gold label. We report overall accuracy for the base and fine-tuned model and, when useful, per-label accuracy (count and accuracy per class). This is the right metric when the eval set is classification-only (e.g. the default sentiment task): it answers ‚Äúhow often does the model get the label right?‚Äù. Per-label breakdown helps spot underperforming classes or imbalance (e.g. ‚Äúneutral‚Äù often harder than ‚Äúpositive‚Äù/‚Äúnegative‚Äù).

**Span match (entity-level)** applies when the eval data contains NER: we treat each `(label, span_text)` pair as one span and count exact matches between gold and predicted entities. We then compute micro-averaged precision (matched predictions / total predictions), recall (matched gold / total gold), and F1. Span-level metrics are stricter than a single ‚Äúright or wrong‚Äù per example: missing one entity in a sentence hurts recall, and an extra wrong span hurts precision. So for NER or multi-task data that includes entities, span P/R/F1 give a clearer picture of extraction quality than accuracy alone.

**Reflection on results.** On the default sentiment-only eval set, only classification metrics are produced. Typically the base model already has some sentiment capability; fine-tuning on our synthetic sentiment data often improves accuracy, with the size of the gain depending on data size, balance, and epoch count. When the eval set includes entities (e.g. after generating NER or NER+classification data), the same pipeline reports span-match metrics: comparing base vs fine-tuned P/R/F1 shows whether training on synthetic NER examples improves extraction. In practice, accuracy is the main number for classification tasks; for extraction tasks, F1 (and the P/R trade-off) should drive model and data choices.

## Limitations and future work

The current implementation has a few clear limitations:

- It only recognises a handful of entity types (person, company, location) and relation types (`works_for`, `lives_in`). Adapting it to other domains would require expanding these banks and templates.
- The label extraction heuristics in the rule-based path are intentionally simple and may still fail on more complex descriptions (‚Äúrate from 1 to 5 stars‚Äù, ‚Äúclassify urgency as low/medium/high/critical‚Äù, etc.), even though the LLM-backed path can often handle these better. In safety-critical settings, you would want additional validation of the LLM‚Äôs JSON output.
- The JSON extraction generator focuses on a single `product` schema. In practice, users may want arbitrary schemas. One improvement would be to parse schema-like descriptions in the task text (for example, `fields: name, storage, price`) and dynamically construct matching JSON structures.
- Although the template banks and combinatorial construction significantly increase variety, the synthetic data is still constrained by manually curated lists. For serious model training, you would want larger, more varied corpora and potentially richer linguistic features, possibly generated or augmented by a local LLM following strict schema constraints.

Despite these limitations, the module provides a clear, GLiNER2-compatible baseline that demonstrates how to infer tasks from descriptions (using either rules or an LLM), compose multi-task outputs, enforce class balance, and integrate with the GLiNER2 training API via JSONL files. The notebook further shows how to fine-tune a GLiNER2 model on balanced, de-duplicated synthetic data and compare it against the base model. Overall, the design is intended as a robust starting point for research engineers to iterate on, rather than an exhaustive data generation system.
